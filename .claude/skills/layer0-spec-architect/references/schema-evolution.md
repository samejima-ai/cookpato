# スキーマ進化プロトコル

データモデル（DB スキーマ、API、メッセージスキーマ、永続イベント）の変更を**破壊的変更なしに**進化させるための規格。
非該当プロジェクト（プロトタイプ、使い捨てツール等）では SPEC.md の「データモデル進化」セクションを省略可。

---

## 適用対象

以下のいずれかに該当する場合、本プロトコルの遵守を強く推奨:

- 本番運用中で既存レコードが存在する
- 外部クライアント（モバイル／SDK／API 利用者）がスキーマに依存する
- ARC が `event-sourcing` に確定している（**完全準拠必須**）
- ARC が `realtime-pubsub` でメッセージスキーマを公開している（推奨）
- ARC が `monolith` でも、本番 DB マイグレーションで無停止デプロイが要件

---

## 後方互換性ポリシー（3 種）

### A. full-compat（完全互換・デフォルト）

新バージョンが旧バージョンの読み書きを完全サポートする。

- 追加は任意（必須でない）カラム／フィールドのみ
- 削除は禁止（非推奨マークのみ）
- 既存フィールドの型変更・セマンティクス変更は禁止
- デフォルト値は旧バージョン読み取り時に適用される

### B. read-compat（読み取り互換）

新バージョンが旧データを読める。旧バージョンは新データを読めない。

- 必須フィールドの追加可（ただし旧データには upcast でデフォルト注入）
- 読み取り時マイグレーション（lazy migration）許容
- ライター側を先にデプロイし、一定期間後にスキーマをクローズ

### C. breaking（破壊的）

双方向互換を放棄する。前提として全クライアントの一斉更新が必要。

- バージョン番号をメジャー更新（v1 → v2）
- 旧バージョンへのリクエストは明示的エラー or 別エンドポイント
- event-sourcing では**採用不可**（既存イベント再生が壊れる）

---

## デプロイ戦略（3 種）

### 1. expand-contract（拡張後収縮）

1. **expand**: 新スキーマを追加。旧スキーマは残す。両方書き込む（dual-write）
2. **backfill**: 既存データを新スキーマに移行
3. **switch-read**: 読み取りを新スキーマに切り替え
4. **contract**: 旧スキーマを削除（十分な待機期間後）

- リスク低。無停止可能
- デフォルトでこの戦略を採る

### 2. blue-green schema

- DB 2 系統を並行運用し、アプリ側でルーティング切替
- 大規模変更時。運用コスト高
- monolith では稀。realtime-pubsub / event-sourcing で pub/sub 層の切替に有効

### 3. versioned-endpoint

- API / イベントにバージョンを埋め込み、複数バージョンを同時提供
- 外部公開 API で旧クライアント切り捨てが不可能なケース
- モバイルアプリとの整合維持で頻出

## event-sourcing での完全準拠

ARC = event-sourcing の場合、以下を**必須**とする（`arc-patterns/event-sourcing.md` の制約とも整合）:

### 必須ルール
- **既存イベントの書き換え・削除は絶対禁止**。訂正は CompensatingEvent を追記
- スキーマ変更は **upcasting**（読み取り時に旧バージョンを新バージョンへ変換）で対応
- イベント型にバージョン番号を必須付与（例: `OrderPlaced.v2`）
- スキーマレジストリ（Confluent Schema Registry / Apicurio 相当）でスキーマを集中管理
- プロジェクション再構築時に全バージョンが解釈可能であること

### 禁止事項
- 既存イベントへの必須フィールド追加（upcast で初期値を注入できる場合のみ許容）
- イベント型の rename（deprecate して新型へ移行）
- event store への DELETE / UPDATE クエリの発行

## マイグレーション手順（SPEC.md 記入例）

```markdown
## データモデル進化

### 現行バージョン
- users テーブル v3（v2 は deprecated、v1 は削除済）

### 互換性ポリシー
- full-compat を原則。breaking 変更は半年ごとのメジャー更新でのみ許容

### デプロイ戦略
- expand-contract を標準。待機期間 14 日

### 進行中のマイグレーション
- users.email_verified 追加（v3 → v4）
  - expand 完了: 2026-04-10
  - backfill 期限: 2026-04-24
  - switch-read 予定: 2026-05-01
  - contract 予定: 2026-05-15
```

## SPEC.md への組み込み（任意セクション）

`meta-spec-template.md` の SPEC.md テンプレートに「データモデル進化」セクションが**任意**で追加される。
以下のいずれかに該当するプロジェクトのみ記入:

- 本番運用中
- 外部クライアントからスキーマ依存がある
- ARC = event-sourcing / realtime-pubsub

該当しないプロジェクトではセクションごと省略してよい（後方互換性を保つデフォルト）。

---

## 検証センサーへの連携

`sensors/computational.md` に以下を含めることを推奨（ARC に応じて増減）:

- スキーマ互換性チェック（ツール例: `schema-registry-compat`, `migra`, `protolock`）
- マイグレーションのドライラン（ステージング DB への適用テスト）

`sensors/inferential.md` に以下:

- 進行中マイグレーションが expand-contract のどの段階にあるか
- 旧バージョンのクライアントが現行 API で壊れないか

---

## プロトコル自己評価

体制事後評価で以下を蓄積:

- breaking 変更が発生した場合、事前に非該当プロジェクト判定をしていたか
- expand-contract の contract を忘れて旧スキーマが残り続けていないか
- event-sourcing プロジェクトで upcast 以外の手段が使われていないか
