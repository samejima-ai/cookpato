# NFR スコア規格

非機能要件（Non-Functional Requirements）を S/U/R と並ぶ第4軸 **N** として定量化するためのスコアリングプロトコル。
業務システム運用・本番サービス型プロジェクトで、体制判定が「何となく M2」に収束する問題を避けるため導入する。

`regime-assessment.md` の S/U/R と同じ判定フローに組み込まれる。

---

## 原則

- NFR は 5 カテゴリ × 0〜3 点で定量化し、合計 0〜15 点を N スコアとする
- 各カテゴリは**存在するかどうか**でまず 0 / 1 点以上を分ける。**強度**に応じて 2〜3 点へ積む
- 曖昧応答は **0 点で開始**する（`REGIME.md` の NFR スコアセクションに記録がない場合のデフォルト = N=0 と同値）
- 4 つのオーバーライドルールで体制判定を上書きする（下記）
- 業界固有のコンプライアンス要件は DOMAIN-CONTEXT.md 側で拾い、N スコアでは「存在する／しない」の粒度に留める

---

## 5 カテゴリの配点

### P: 性能（Performance）

| スコア | 条件 |
|---|---|
| 0 | レイテンシ・スループット要件なし |
| 1 | 体感品質（秒単位の応答）のみ明示 |
| 2 | レイテンシ SLO（p95 〜 ms/s）が数値指定されている |
| 3 | 高スループット（大量同時接続・大容量データ処理）＋レイテンシ両立が要件 |

### A: 可用性（Availability）

| スコア | 条件 |
|---|---|
| 0 | 落ちても困らない（実験・自分用） |
| 1 | 平日日中のみの内部ツール水準 |
| 2 | 99.9% 相当、業務時間内停止が痛い |
| 3 | 99.95% 以上 or 24/7 運用、停止が金銭・安全影響 |

### S: セキュリティ（Security）

| スコア | 条件 |
|---|---|
| 0 | 認証不要・公開情報のみ |
| 1 | ログイン必須、内部情報を扱う |
| 2 | PII / 個人情報 / 決済情報 / 医療情報のいずれかを扱う |
| 3 | 規制業界（金融・医療・公共）＋外部攻撃面あり |

### C: コンプライアンス（Compliance）

| スコア | 条件 |
|---|---|
| 0 | 特別な規制なし |
| 1 | 社内規定・業界ガイドライン程度 |
| 2 | 法令準拠（個人情報保護法・業法）が明示要件 |
| 3 | 複数規制の同時準拠（例: 個人情報保護法 + 金商法 + 業界固有）or 監査必須 |

### L: スケーラビリティ（Scalability）

| スコア | 条件 |
|---|---|
| 0 | 単一環境・固定規模で足りる |
| 1 | 将来の拡張を意識するが現時点は単一構成 |
| 2 | 水平スケール前提、負荷変動吸収が要件 |
| 3 | マルチリージョン・グローバル分散・バースト対応が要件 |

---

## N スコアの合計と意味

合計値の範囲: **N = 0 〜 15**

| N 範囲 | 典型像 |
|---|---|
| 0〜2 | 非機能要件ほぼなし。実験・私用・社内ツール初期 |
| 3〜4 | 内部運用ツール、停止や漏洩が業務影響あるレベル |
| 5〜8 | 業務システム運用・本番サービス。運用設計が必要 |
| 9〜15 | ミッションクリティカル。アーキテクチャ選定段階から NFR 駆動 |

N は**判定を縛るのではなく、上書きルールの発動条件**として使う。単独で M2→L2 等の強制はしない。

---

## オーバーライドルール（4 条件）

N 軸に由来する体制判定の強制上書き。`regime-assessment.md` のオーバーライド優先順位表に組み込まれる。

### ルール1: N ≥ 9 → モノリス解除を強制

- ARC デフォルト `monolith` を **強制解除**し、`arc-patterns/` 配下から人間に選択させる
- N ≥ 9 はミッションクリティカル帯。モノリス単独では可用性・スケーラビリティの両立が難しいため
- 選択は人間の判断献上で行う（AI は推奨のみ）

### ルール2: N ≥ 5 → 推奨切替検討（2段階化）

- **ARC 選択を人間に提案**する（強制ではない）
- S/U/R 一次判定が M1 でも **M2 以上を推奨**する（強制は既存 R≥2 ルール側）
- 運用設計の必要性をドキュメントで指摘する（DONT.md に L3 運用層を先に置かないことを明記）

### ルール3: セキュリティ ≥ 2 → M2 強制

- S (security) カテゴリが 2 点以上なら **M2 以上を強制**（M1 不可）
- 独立検証（layer1-independent-reviewer）を必須化するため
- 既存の R ≥ 2 → M2 強制ルールと**競合しない**（両方発火しても結果は M2 で一致）
- ただし、R=1 かつ S=2 の組合せで本ルールが単独発火する（金銭関与なしでも PII ありなら M2 強制）

### ルール4: 可用性 = 3 → 権限レベル記録必須

- A (availability) カテゴリが 3 点（24/7・ミッションクリティカル）なら REGIME.md に**権限レベルの明示記録を必須**化する
- デフォルト `L0-2`（`permission-delegation.md` 参照）。明示しないと REGIME.md が不完全と判定される
- `permission-delegation.md` の昇格条件と齟齬なし

## オーバーライド優先順位（N 軸を組み込んだ完全版）

`regime-assessment.md` 既存の優先順位表に N 系を追記した統合版:

1. **L2 発動閾値** → L2 確定（最優先）
2. **U ≥ 3** → L0 対話延長（モード確定保留）
3. **N ≥ 9** → モノリス解除強制（ARC 選択を人間判断献上）
4. **N ≥ 5** → ARC 推奨切替 + 2段階化推奨（強制ではない）
5. **R ≥ 2** → M2 以上強制
6. **セキュリティ ≥ 2** → M2 以上強制
7. **可用性 = 3** → 権限レベル記録必須化

ルール 5・6 は同じ結果（M2 強制）に収束するため衝突しない。ルール 3 は ARC を上書きするが M2/L2 判定自体は上書きしない。

---

## 判定フロー（N 軸追加後）

```
1. 対話から S, U, R, N を算出
2. L2発動閾値を超える？
   YES → L2 確定
3. U ≥ 3 ?
   YES → L0 対話延長
4. N ≥ 9 ?
   YES → ARC モノリス解除。arc-patterns/ から人間選択を献上
5. N ≥ 5 ?
   YES → ARC 選択を推奨、2段階化を検討
6. S + U + R で一次判定（0〜3=M1 / 4〜=M2）
7. R ≥ 2 ? または N の security ≥ 2 ?
   YES → M2 に引き上げ
8. N の availability = 3 ?
   YES → 権限レベル明示記録を REGIME.md に必須化
9. 確定 → REGIME.md 出力
```

## REGIME.md への記録フォーマット

```markdown
## NFRスコア（N）

- 性能 P: [0-3]
- 可用性 A: [0-3]
- セキュリティ S: [0-3]
- コンプライアンス C: [0-3]
- スケーラビリティ L: [0-3]
- 合計 N: [0-15]

## NFRオーバーライド適用

- N ≥ 9（モノリス解除強制）: [適用 / 非適用]
- N ≥ 5（ARC切替推奨）: [適用 / 非適用]
- セキュリティ ≥ 2（M2強制）: [適用 / 非適用]
- 可用性 = 3（権限レベル記録必須）: [適用 / 非適用]
```

## デフォルトと後方互換

- **`REGIME.md` に NFR スコアセクションがないプロジェクト**は **N=0** として扱う（全オーバーライド非適用）
- 既存 REGIME.md 形式（S/U/R のみ）のプロジェクトはそのまま通る
- 本規格は**新規プロジェクトと、機能追加時に NFR 情報が引き出せた既存プロジェクトのみに適用**する
- Layer 0 対話で N 軸の情報が引き出せなかった場合は N=0 を明示記録し、次サイクル以降で精緻化する

---

## dialog-questions.md との連携

非エンジニアから NFR 情報を引き出す質問例は `dialog-questions.md` 末尾の NFR セクションに記載する。
技術用語（SLA / SLO / PII 等）を使わず、「壊れたらどれくらい困るか」「どんな人のデータを扱うか」等の日常語で聞く。

## プロトコル自己評価

体制事後評価（`delivery-format.md`）で N 軸の妥当性も蓄積する:

- 過剰：N=6 で M2 強制したが独立検証が無指摘だった → N の重みを見直し
- 過少：N=3 で M2 通過したが運用で事故った → 判定表にカテゴリ追加
- オーバーライド妥当性：N ≥ 9 のモノリス解除が実際に必要だったか
