# 対話質問例集

Layer 0（spec-architect）が非エンジニアから体制判定情報を引き出すための質問例。
技術用語を使わず、人間が直感的に答えられる形で設計する。

---

## 原則

- 技術用語を避ける（「SLA」「認証」「永続化」等は使わない）
- 具体的な数字や選択肢を提示する
- 1回のターンで質問を詰め込まない（最大2〜3問）
- 「分からない」と答えられる余地を残す
- 仕様の具体化と自然に織り交ぜる

---

## 規模スコア（S）を引き出す質問

### 機能数の把握

避ける：「機能はいくつですか？」

推奨：
- 「このツールで『できること』を思いつく限り挙げるとどんな感じ？」
- 「『〜できたらいいな』と思うことを箇条書きにするとどれくらいになりそう？」
- 「1〜3個くらい？ 4〜10個くらい？ それ以上？」

### 画面数の把握

避ける：「画面構成は？」「UIは？」

推奨：
- 「使う人が見る画面って、どんな画面がありそう？」
- 「設定画面とメイン画面、みたいに分かれてる？」
- 「画面が必要なやつ？ それとも裏で動くだけ？」

### 登場人物の把握

避ける：「ユーザーロールは？」「権限は？」

推奨：
- 「使う人はどんな人？ 自分だけ？ 他の人も使う？」
- 「管理する人と使う人って分かれてる？」
- 「全員同じことができればいい？ それとも人によって見える範囲が違う？」

### 外部連携の把握

避ける：「API統合は？」「外部サービスは？」

推奨：
- 「他のサービスとつながる必要ある？ 例えばGmailとかSlackとか」
- 「データをどこかから取ってくる？ どこかに送る？」
- 「全部このツールの中で完結する？」

---

## 不確実性スコア（U）を引き出す質問

### イメージの明確さ

避ける：「仕様は固まってますか？」

推奨：
- 「頭の中でどれくらい絵が見えてる？ 具体的？ ぼんやり？」
- 「完成したときの画面、思い浮かべられる？」
- 「『こういう感じ』って他の人に説明できる？ それともまだ探り探り？」

### 既知度

避ける：「技術選定は？」「類似システムは？」

推奨：
- 「似たようなもの、使ったことある？」
- 「『こういうの欲しい』って思ったきっかけは、何か似たサービス？」
- 「全く初めての領域？ それとも普段触ってるジャンル？」

### 変更予測

避ける：「ロードマップは？」「保守計画は？」

推奨：
- 「1回作って終わり？ それとも使いながら直していく感じ？」
- 「完成したら放置？ それとも機能追加していく予定？」

---

## リスクスコア（R）を引き出す質問

避ける：「SLAは？」「可用性要件は？」「セキュリティ要件は？」

推奨：
- 「壊れたらどれくらい困る？」
- 「使うのは自分だけ？ 他の人にも渡す？」
- 「お金が絡む？ それとも趣味・実験の範囲？」
- 「動かなくなっても笑える？ それとも困る？」

## 対話パターン例

### パターン1：最初の一言で情報が薄いケース

ユーザー：「タブマネージャーみたいなChrome拡張を作りたい」

対話の流れ：
1. まず目的と全体像を確認
   「どんな悩みを解決したい感じ？」
2. 機能の広がりを掴む
   「『できること』を思いつく限り挙げてみて」
3. リスク確認
   「これ、自分だけ使う？ Chrome Web Storeに公開する感じ？」
4. 画面の有無
   「設定画面みたいなのはいる？」

このパターンで S=3, U=1, R=2 程度の情報が集まる。
合計 6点 + R≥2 → **M2 標準モード**（L1 + layer1-independent-reviewer）で運用。

### パターン2：「なんとなく欲しい」だけのケース

ユーザー：「SaaSっぽいの作りたい」

対話の流れ：
1. まず具体化を試みる
   「どんな悩みを解決したい？ 誰の悩み？」
2. 既知度を探る
   「似たサービス使ってて『こうだったらいいのに』って思った？」
3. 答えが出ないなら不確実性を認める
   「じゃあまずイメージを一緒に固めるところから始めよう」

このパターンで U≥3 を検出した時点で **L0 対話延長**を宣言。
モード確定は保留し、仕様が煮詰まったら S/U/R を再算出する。
早く形が欲しい場合は暫定 M1 で小さく試作し、使いながら仕様を固めるのも可。

### パターン3：鮮明だが規模が読めないケース

ユーザー：「社内のタスク管理ツール作りたい」

対話の流れ：
1. 画面構成を掴む
   「使う人が見る画面は？ 一覧とか詳細とか」
2. 登場人物を掴む
   「使う人は何種類？ 管理者と一般メンバー、みたいな？」
3. 外部連携を掴む
   「SlackとかGmailとつなぐ必要ある？」
4. リスク確認
   「社内で本番運用する感じ？ 試作？」

このパターンで S=5〜7, R=2〜3 の範囲が見えてくる。
- S=5〜6 かつ L2閾値未満 → **M2 標準モード**
- S=7以上 + 独立ドメイン数≥5（ユーザー管理/タスク/通知/レポート/権限 等）→ **L2 発動**を検討
- 独立ドメイン数が閾値未満なら M2 で吸収

## 質問の順序

体制判定情報を取得する順序は以下を推奨：

1. **目的・やりたいこと**（仕様具体化と共通）
2. **機能の広がり**（規模S）
3. **登場人物**（規模S）
4. **画面の有無**（規模S）
5. **既知度・類似物の経験**（不確実性U）
6. **イメージの明確さ**（不確実性U。4までの答え方から推測も可能）
7. **リスク**（リスクR。最後に確認）
8. **外部連携**（規模S。リスクと絡めて聞くと自然）
9. **変更予測**（不確実性U。最後でよい）

後半の項目は前半の回答から推測できる場合は質問を省略する。
判定プロトコルの完全性より対話の自然さを優先する。

## 振り返り儀式の質問テンプレ（Lifecycle ≥ 1 で使用）

履歴層 `history/` がある既存プロジェクトで、L0 対話の冒頭に実行する振り返り儀式のテンプレ。
儀式プロトコル全体は `ritual-protocol.md` を参照。

### F1: 過去文脈サマリ提示（質問は最大1問）

SUMMARY.md を提示した上で：

- 「前回ここまで進みました（要約）。今回はこれの続きで合ってますか？」

それ以上は聞かない。F1 の目的は認識共有のみ。

### F2: 認識ズレ検出（検出時のみ質問）

AI が過去 INTENT と現欲求を照合し、以下のパターンを検出した場合のみ質問する。

**矛盾検出時**：

- 「F001 の『〇〇』と今回の『△△』は両立しないです。どちらを優先しますか？」

**廃止機能の復活要求検出時**：

- 「F〇〇〇 は YYYY-MM-DD に『〜』を理由に廃止しました。復活条件を満たしますか？」

**却下案の再提案検出時**：

- 「〇〇案は当初『〜のリスク』で却下しました。状況変化はありますか？」

検出なしなら F2 は無言で通過。

### F3: 履歴更新予告（通知形式、承認のみ求める）

質問ではなく通知：

- 「以下を INTENT.md に追加します: [内容]」
- 「F001 の条件を v2 に更新します: [差分]」
- 「R=2 への変更により REGIME.md を更新します」

人間の反応は「承認 / 修正指摘」のいずれか。修正指摘があれば該当箇所のみ直して再提示。

### 曖昧応答が返った場合（E1対応）

人間が「たぶん」「どうだったかな」「どっちでもいい」等で返した場合、**追加質問をしない**。
AI が過去履歴から最適解を推定し、通知形式で充当する：

- 「〇〇と推定しました。違ったら次回教えてください」

INTENT.md の該当項目に `**確度**: AI推定 (YYYY-MM-DD)` を付与し、次回儀式 F2 で一括確認する。
曖昧応答が2回連続したら儀式レベルを自動 -1 格下げ。

### 儀式拒否への対応（E2対応）

「儀式いらない、すぐ作って」「軽くやりたい」等の拒否発話を検出したら、儀式を**完全スキップ**し REGIME-LOG.md に記録する。
質問を重ねず、L1 に引き継ぐ（L1 が背景で履歴照合を回す）。
拒否が5回連続したら SUMMARY.md に警告常設（人間裁量は最後まで尊重、強制介入なし）。

## NFR（非機能要件）質問例

技術用語（SLA / SLO / PII 等）は避け、日常語で聞く。詳細は `nfr-scoring.md` を参照。

### 性能（P）
- 「ボタン押してから反応が何秒以内じゃないと困る感じですか？」
- 「同時に使う人は何人くらい想定？ピーク時と普段の差は？」

### 可用性（A）
- 「壊れて使えない時間があると、どれくらい困りますか？ 1日？ 1時間？ 1分もダメ？」
- 「夜中や休日も動いている必要がありますか？」

### セキュリティ（S）
- 「どんな人のデータを扱いますか？ 社員？ お客様？ 個人名・住所・お金は含む？」
- 「このシステムが外部から攻撃されたら、何が漏れたり壊れたりしますか？」

### コンプライアンス（C）
- 「法律や業界ルールで『こうじゃないとダメ』というのはありますか？」
- 「監査やチェックを受ける必要がありますか？ 誰から？」

### スケーラビリティ（L）
- 「使う人や扱うデータが 10 倍、100 倍に増えたら同じ仕組みで動きますか？」
- 「海外からも使いますか？ 国内だけ？」

## ARC（アーキテクチャ）質問例

詳細は `arc-patterns/` 配下 3 パターンを参照。

- 「リアルタイムで通知が届く／ライブで状態が更新される、みたいなのは必要？」
- 「『あの時点の状態を復元したい』みたいな監査／タイムトラベル要件は？」
- 「同時接続ユーザ数は何人くらいの想定？」

NFR スコアと合わせて `monolith / realtime-pubsub / event-sourcing` から推奨提示。最終決定は**人間判断献上**。

## ドメイン文脈質問例

詳細は `domain-context-dialog.md` を参照。該当なしなら DOMAIN-CONTEXT.md ごと省略可。

### Cat-1 業界・業務
- 「どんな業界／業務のプロジェクトですか？ 社内向け？ 顧客向け？」
- 「類似サービスで参考にしてるものはありますか？」

### Cat-2 業務慣習
- 「この業務特有の『外の人には分かりにくいルール』はありますか？」
- 「業界用語で、他業界と意味が違う言葉は？」

### Cat-3 既存システム
- 「今使っているシステム／ツールは？ 置き換え？ 併用？」
- 「データ連携が必須な外部システムは？ 接続方式は？（API / CSV / FAX）」

### Cat-4 規制
- 「法令・業法で準拠必須なものは？（個人情報保護法 / 下請法 / 業界固有法）」
- 「契約書・取引記録の保管期間は？」

### Cat-5 用語集
- 「社内や業界でよく使う略語・通称は？ 正式名称と違うものは？」

## 権限レベル質問例

詳細は `permission-delegation.md` を参照。

- 「AI が勝手に git commit するのは OK？ push は？ デプロイは？」
- 「外部 API の課金・契約判断は人間でやりたい？ AI に任せても OK な上限は？」
- 「壊れたら困る本番環境は何ですか？ そこは絶対に人間判断にしたい？」

デフォルトは L0-2（最小権限）。L0-3 昇格は Layer 0 対話で明示同意を得た場合のみ。
