# 献上物フォーマット定義

Layer 1が人間に提出する献上物の構造と記述ルール。

---

## 献上物タイプA：仕様レビュー結果

仕様レビューで問題を検出し、開発に入れない場合に提出する。

### 構造

```markdown
# 仕様レビュー結果

## ステータス: 開発不可

## 検出された問題

### [問題1のタイトル]
- **分類**: 不足 / 矛盾 / スコープ侵食 / 実装不可能 / 体制齟齬
- **該当箇所**: SPEC.md#[セクション名] または REGIME.md
- **詳細**: [具体的に何が問題か]
- **影響**: [この問題を放置した場合に発生する具体的な不具合]
- **提案**: [Layer 0での解決案。可能であれば選択肢を提示]

### [問題2のタイトル]
...

## 開発可能な範囲（任意）
[問題のある機能を除外すれば残りは開発可能な場合、その旨を記載]
```

### 記述ルール
- 問題は重大度の高い順に記載する
- 「問題がある」だけでなく「どうすれば解決するか」の提案を必ず添える
- Layer 0ドキュメントのどの箇所を修正すべきかを明示する
- 体制齟齬（REGIME.md のモードが実態と合わない）を検出した場合も本フォーマットで報告する

---

## 献上物タイプB：成果物

開発完了時に提出する。動作するコードとフィードバックレポートのセット。

### 構造

```
delivery/
├── DELIVERY.md          # 献上レポート（全モード必須）
├── VERIFICATION.md      # 独立検証レポート（M2以上で必須）
├── INTEGRATION.md       # 統合レポート（L2のみ。L2オーケストレータが生成）
├── HISTORY-DIFF.md      # 履歴層更新差分（Lifecycle ≥ 1 で必須）
├── src/                 # ソースコード
├── tests/               # テストコード
└── (その他プロジェクトファイル)
```

### DELIVERY.md テンプレート

```markdown
# 献上レポート

## ステータス: 完了 / 一部未完了

## 体制情報

- **モード**: M1 / M2 / L2
- **REGIME.md参照日**: [日付]
- **判定時のAI能力バージョン**: [REGIME.md から転記]
- **権限レベル**: L0-2 / L0-3（REGIME.md から転記、未記載時は L0-2）
- **ARC**: [monolith / realtime-pubsub / event-sourcing]（未指定時は monolith）

## 実装済み機能

- [機能A]: 完了
- [機能B]: 完了
- [機能C]: 一部完了（詳細は下記）

## 自己検証結果

### 計算的センサー
- ビルド: PASS / FAIL
- 型チェック: PASS / FAIL
- リンター: PASS / FAIL
- テスト: PASS / FAIL ([通過数]/[総数])

### 推論的センサー
- 仕様合致: PASS / FAIL
- 動作確認: PASS / FAIL
- 使用確認: PASS / FAIL

### 独立検証（M2以上で必須）
- 検証レポート: VERIFICATION.md参照
- 最終判定: PASS / FAIL

## 仕様フィードバック（任意）

[開発中に発見した仕様の改善提案。Layer 0への逆流情報]

### [フィードバック1のタイトル]
- **該当箇所**: SPEC.md#[セクション名]
- **発見内容**: [開発中に気付いた仕様の曖昧さ・改善余地]
- **AIの解釈**: [今回の実装でどう解釈したか]
- **提案**: [仕様をどう修正すると次回以降の品質が上がるか]

## 未解決事項（任意）

[自力修正を試みたが解決できなかった問題]

### [未解決事項1]
- **内容**: [何が問題か]
- **試行した対策**: [何を試したか]
- **失敗理由**: [なぜ解決できなかったか]

## 体制事後評価（必須）

### 体制妥当性の自己評価

- **判定モード**: M1 / M2 / L2
- **実績から見た妥当性**: 過剰 / 妥当 / 過少
- **評価根拠**: [下記チェックリスト参照]

### 評価チェックリスト

#### 過剰判定の兆候
- [ ] layer1-independent-reviewer が何も指摘しなかった（M2以上）
- [ ] ドメイン間通信がほぼ発生しなかった（L2）
- [ ] 自力修正ループが1回も発生しなかった
- [ ] 検証に要した時間が実装時間を大きく上回った

該当多数 → 次回は一段階下のモードで足りる可能性

#### 過少判定の兆候
- [ ] 自力修正ループが3回上限に達した
- [ ] 未解決事項が2件以上発生した
- [ ] 仕様レビューで軽微な不足として流したが、実装で困った
- [ ] コンテキスト窓の圧迫を感じた（M2で仕様が収まりきらない等）
- [ ] 複数の独立ドメインが暗黙に発生し、整合性取りに苦労した

該当多数 → 次回は一段階上のモードが必要（M1→M2, M2→L2）

### スコア事後調整の提案

[該当する場合のみ記載]

- 規模スコア（S）: 当初X点 → 実績Y点
- 不確実性スコア（U）: 当初X点 → 実績Y点
- リスクスコア（R）: 当初X点 → 実績Y点
- L2発動閾値の超過: [実際に超えた項目があれば記載]

### 判断献上履歴（C1 発動記録）

[開発中に `permission-delegation.md` の C1 発動が起きた場合のみ記録]

- [Cat-A〜Cat-E カテゴリ]: [内容] → [人間の回答・採用案]
- ...

### 割り込み履歴（C2 発動記録）

[開発中に人間からの割り込みが発生した場合のみ記録]

- [タイミング]: [内容] → [対応]
- ...

### 次回プロジェクトへの申し送り

[類似プロジェクトでモード判定・権限レベル運用を改善するための知見]

## 起動方法

[成果物を確認するための手順]
```

### 記述ルール
- 実装済み機能はSPEC.mdの機能一覧と1:1で対応させる
- 自己検証結果は全センサーの結果を記載する（省略しない）
- 仕様フィードバックは強制ではない。発見した場合のみ記載する
- 未解決事項には必ず「試行した対策」を含める（「できませんでした」だけにしない）
- **体制事後評価は必須**。過剰／妥当／過少のいずれかを明記する

## VERIFICATION.md（M2以上で必須）

layer1-independent-reviewer が生成するレポート。L1からの起動結果として出力される。

### テンプレート

```markdown
# 独立検証レポート

## 検証実行日
[日付]

## 検証エージェント情報
- 役割: 独立レビュアー（実装コンテキスト隔離）
- 入力: SPEC.md, DONT.md, REGIME.md, sensors/inferential.md, 成果物のみ
- 参照しないもの: L1の作業メモ・中間思考・試行錯誤ログ

## 検証結果

### 仕様合致
| 機能 | 判定 | 備考 |
|---|---|---|
| [機能A] | PASS / FAIL | [備考] |
| [機能B] | PASS / FAIL | [備考] |

### 動作確認
- 起動: PASS / FAIL
- 主要操作パス: PASS / FAIL
- エラーハンドリング: PASS / FAIL

### 発見事項

[独立視点で発見した問題・懸念]

## 最終判定

PASS / FAIL

[FAIL時は実装エージェントへの差戻し理由を明記]
```

## INTEGRATION.md（L2のみ）

layer2-integration-verifier が生成する統合レポート。L2オーケストレータが最終献上に含める。
**L1単独では生成しない**。L2発動時にのみ存在する。

### テンプレート

```markdown
# 統合レポート

## ドメイン構成
- ドメインA: [概要] / モード M1 or M2
- ドメインB: [概要] / モード M1 or M2
- ...

## 各ドメインVERIFICATION.md の確認
[全ドメインがPASSであることの確認]

## インターフェース整合性

| 境界 | 契約 | 整合性 |
|---|---|---|
| A→B | [インターフェース定義] | PASS / FAIL |
| B→C | [インターフェース定義] | PASS / FAIL |

## 全体不変条件
[SPEC.md のクロスドメイン制約の検査結果]

## E2E統合テスト結果

[統合テストの実行結果]

## 調停履歴

[不整合検出時の判断と修正要請の記録]

## L2レベルの体制事後評価
- **L2発動の妥当性**: 過剰 / 妥当 / 過少
- **サブドメイン分割の妥当性**: 過剰分割 / 妥当 / 分割不足
- **次回申し送り**: [類似規模プロジェクトでのL2判定改善のための知見]

## 最終判定

PASS / FAIL
```

## HISTORY-DIFF.md（Lifecycle ≥ 1 で必須）

既存プロジェクト（L=1/L=2）の献上時に、L1 が生成する履歴層更新差分。
人間は承認レベル（A/B/C）ごとにレビューする。

### テンプレート

```markdown
# 履歴層更新差分

## Lifecycle
- 現在の Lifecycle: [L=1 / L=2]
- 儀式レベル: [0 / 1 / 2 / 3]（L0 儀式で採用された値、スキップ時は「スキップ」）
- 儀式スキップの有無: [あり（E2 拒否） / なし]

## レベルA（自動承認・適用済み）

適用を完了。人間確認は不要。

### CHANGELOG.md 追記
- [YYYY-MM-DD] [機能名] [追加/更新/廃止] — [概要]

### SUMMARY.md 再生成
- 更新トリガー: L1献上
- 差分: [主要変更点の箇条書き]

### PATTERNS.md 追記（発見時のみ）
- [罠/成功パターンの名称] — [内容]

## レベルB（確認推奨・通知のみ）

デフォルト承認。異議があれば指摘してください。

### INTENT.md 新規追加
- **F[番号]**: [機能名]
  - **意図**: [なぜこの機能か]
  - **条件**: [主要条件]
  - **優先度**: [高/中/低]
  - **確度**: 人間確定 / AI推定 (YYYY-MM-DD)

### REGIME-LOG.md 更新
- 判定 vs 実績の対応: [過剰/妥当/過少]
- 儀式記録: レベル[N]、検出件数[M]、スキップ有無

### ARCH-DECISIONS.md 追記（設計判断があった場合）
- **AD[番号]**: [タイトル]
  - コンテキスト / 選択肢 / 決定 / 根拠

## レベルC（必須承認・未適用）

人間の承認がない限り適用しません。

### INTENT.md 廃止・訂正
- **F[番号]** 廃止提案: [理由・代替機能]
- **F[番号]** 訂正: ~~旧条件~~ → 新条件（理由: [説明]）

### REGIME.md スコア変更
- S: X → Y（根拠: [説明]）
- R: X → Y（根拠: [説明]）

### 確度メタデータ上書き
- F[番号] の AI推定を確定値に変更

## 履歴整合性チェック（儀式スキップ時 or 検出時のみ）

L0 儀式が E2 拒否でスキップされた場合、L1 が実装後に背景照合を実行。
検出なしなら本セクションは「検出なし」と記載するのみ。

### 矛盾検出
- [過去INTENT F[番号] との矛盾]: [詳細]

### 廃止機能への回帰
- [廃止済み F[番号] の再実装痕跡]: [詳細]

### 却下案の再実装
- [却下案リスト X との一致]: [詳細]

## archive 予定

2年経過した廃止 INTENT を `history/archive/` へ移動。
- [F番号] [機能名] 廃止日 YYYY-MM-DD → archive へ
```

### 記述ルール

- レベル分類を間違えない。廃止・訂正は必ずレベルC
- AI推定で埋めた確度メタデータは必ず次回 F2 で確認する旨を明記
- 履歴整合性チェックは検出があった場合のみ詳細記述、なしなら1行
- archive 年数は REGIME.md の「archive 移動年数」設定に従う

## 事後評価の蓄積活用

DELIVERY.md の体制事後評価セクションは、複数プロジェクトを通じて蓄積することで
以下のメタ知見を得られる：

- **判定プロトコルの精度**：過剰判定率・過少判定率の把握
- **スコアリング表のキャリブレーション**：どの小項目の配点が過大／過小か
- **モード境界の再検討**：M1⇔M2 の境界閾値調整、L2発動閾値の見直し
- **AI能力バージョン別の傾向**：モデル更新でどの閾値が緩和可能になったか
- **ユーザー別の傾向**：仕様確定度を自己評価する癖のズレ補正

これらは Layer 0 の `references/regime-assessment.md` に反映し、運用知として育てていく。
特に **AI能力バージョン** 記録は将来モデルでの閾値緩和判断に必須。
