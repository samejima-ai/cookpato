# 献上物フォーマット定義

Layer 1が人間に提出する献上物の構造と記述ルール。

---

## 献上物タイプA：仕様レビュー結果

仕様レビューで問題を検出し、開発に入れない場合に提出する。

### 構造

```markdown
# 仕様レビュー結果

## ステータス: 開発不可

## 検出された問題

### [問題1のタイトル]
- **分類**: 不足 / 矛盾 / スコープ侵食 / 実装不可能
- **該当箇所**: SPEC.md#[セクション名]
- **詳細**: [具体的に何が問題か]
- **影響**: [この問題を放置した場合に発生する具体的な不具合]
- **提案**: [Layer 0での解決案。可能であれば選択肢を提示]

### [問題2のタイトル]
...

## 開発可能な範囲（任意）
[問題のある機能を除外すれば残りは開発可能な場合、その旨を記載]
```

### 記述ルール
- 問題は重大度の高い順に記載する
- 「問題がある」だけでなく「どうすれば解決するか」の提案を必ず添える
- Layer 0ドキュメントのどの箇所を修正すべきかを明示する

---

## 献上物タイプB：成果物

開発完了時に提出する。動作するコードとフィードバックレポートのセット。

### 構造

```
delivery/
├── DELIVERY.md          # 献上レポート
├── src/                 # ソースコード
├── tests/               # テストコード
└── (その他プロジェクトファイル)
```

### DELIVERY.md テンプレート

```markdown
# 献上レポート

## ステータス: 完了 / 一部未完了

## 実装済み機能
- [機能A]: 完了
- [機能B]: 完了
- [機能C]: 一部完了（詳細は下記）

## 自己検証結果

### 計算的センサー
- ビルド: PASS / FAIL
- 型チェック: PASS / FAIL
- リンター: PASS / FAIL
- テスト: PASS / FAIL ([通過数]/[総数])

### 推論的センサー
- 仕様合致: PASS / FAIL
- 動作確認: PASS / FAIL
- 使用確認: PASS / FAIL

## 仕様フィードバック（任意）
[開発中に発見した仕様の改善提案。Layer 0への逆流情報]

### [フィードバック1のタイトル]
- **該当箇所**: SPEC.md#[セクション名]
- **発見内容**: [開発中に気付いた仕様の曖昧さ・改善余地]
- **AIの解釈**: [今回の実装でどう解釈したか]
- **提案**: [仕様をどう修正すると次回以降の品質が上がるか]

## 未解決事項（任意）
[自力修正を試みたが解決できなかった問題]

### [未解決事項1]
- **内容**: [何が問題か]
- **試行した対策**: [何を試したか]
- **失敗理由**: [なぜ解決できなかったか]

## 起動方法
[成果物を確認するための手順]
```

### 記述ルール
- 実装済み機能はSPEC.mdの機能一覧と1:1で対応させる
- 自己検証結果は全センサーの結果を記載する（省略しない）
- 仕様フィードバックは強制ではない。発見した場合のみ記載する
- 未解決事項には必ず「試行した対策」を含める（「できませんでした」だけにしない）
